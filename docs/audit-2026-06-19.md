## 3. Performance Baseline

### Database Query Performance

**Query**: `scopeNearby` + `byPopularity` with SF coordinates, 25km radius.

```
SELECT *, (6371 * acos(...)) AS distance
FROM restaurants
WHERE latitude IS NOT NULL AND longitude IS NOT NULL
  AND (6371 * acos(...)) <= 25
ORDER BY popularity_score DESC
LIMIT 20
```

**EXPLAIN QUERY PLAN** (SQLite):
```
SCAN restaurants
USE TEMP B-TREE FOR ORDER BY
```

**Findings**:
- 🔴 **Full table scan** — No index on `latitude`/`longitude` for proximity filter
- 🔴 **Temp B-tree for sort** — No index on `popularity_score` for ORDER BY
- ✅ **Acceptable for 23 rows** — Current DB size makes scan fast enough
- ⚠️ **Won't scale** — At 1K+ restaurants, latency will degrade significantly

**Latency** (local, sqlite): <50ms for 23 restaurants

## Recommendations

### High Priority

1. **Add geospatial indexes** — Spec 008 addresses this
   - Add composite index on `(latitude, longitude)` or use R-tree for proximity
   - Add index on `popularity_score` for sorting

2. **Remove Overpass mirrors 2 & 3** — They're timing out
   - Keep only `https://overpass-api.de/api/interpreter`
   - Or increase timeout to 60s and retry less aggressively

3. **Parallelize live fetches** — Spec 009 addresses this
   - Current: BizData → Overpass (sequential, ~15s)
   - Target: Concurrent fetches (~5s)

### Medium Priority

4. **Cache warming** — Pre-populate cache for common SF coordinates
   - Reduce cold start latency for first user

5. **Award seeding** — Add some Michelin-awarded restaurants
   - Test `has_award` signal in scoring

6. **Monitor BizData latency** — 11.6s is high for a single query
   - Investigate if BizData can be optimized or if it's a baseline limit
